    This is a line graph of Selection Sort and is almost identical to Insertion sort because they are both very similar in the way that they sort data. Insertion sort goes one element at a time and inserts the lower value in front of the higher value and goes through many stages of unsorted arrays. Selection sort selects the smallest element one element at a time and moves it to the right place and searches the remaining data, this is slightly less efficient than insertion sort but took much less time than the Insertion Sort in my trials. Values are continuously checked to see if they are less than the value of the previous element, and keep being swapped if true It goes through mutile runs through the data till its fully sorted. The reason all of these graphs look similar from the best case of ordered and worst the reverse sort is because in all three cases the time complexity of selection sort is O(n*2).

This displays how inneficcent bubble sort is for large lists but it is very efficient for small amounts of words. You should only use bubble sort for lists under 1000 words and for more efficiency below 100 would be best because bubble sort individually compares each word and takes much longer than insertion or selection sort. Hypothesis


    All the graphs are almost identical for 1-1000 words and you can see the real difference after 1K, all the graphs have an exponential increase after those thousand words. The only sort that is recursive is Alphabetical Sort which sorts a lot faster than any of the other sorting algorithms because of that. The two stable sorts are Bubble sort and insertion sort which make the equivalent elements retain their relative positions, after sorting. Stable sorts are generally slower than unstable sorts which are proved in the table when you compare bubble and insertion sort times to the selection and alphabetical sort. The speed of a sort is determined by the number of passes, it takes to sort some sorts are more efficient for a smaller amount of words and others take less time for larger amounts, like our alphabetical sort.The selection sort is the only algorithm among these four to have a best case scenario time complexity of O(n^2) which is not even close in comparison of my sort witch has a time complexity of worst case running time of O(n+b) and best case T(n) = d(n+b0). The sort that we created is much better in comarisosn to these standard sort as proved in the timings of the tests and the time complexity of the sorts.
